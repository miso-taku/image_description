# 要件定義書

## 1. インテント分析

- **ユーザー要求**: Python 3.12.8、uv、FastAPI、PydanticAI、Next.js、React、OpenAI APIを用いて、入力画像を日本語で説明するWebアプリケーションを構築する。
- **要求種別**: 新規プロジェクト
- **スコープ**: バックエンド、フロントエンド、OpenAI API連携を含む複数コンポーネント
- **複雑度**: 中程度
- **要件深度**: 標準

## 2. プロダクト目標

ユーザーがローカル環境で画像を1枚選択し、説明の詳しさを指定して、画像内容を日本語の自然文として安全かつ分かりやすく取得できるアプリケーションを提供する。

## 3. 対象ユーザー

- 画像の内容を文章で素早く把握したい一般ユーザー
- 画像説明生成を試したい開発者
- 初期リリースではローカル環境の利用者のみを対象とする

## 4. 機能要件

### FR-01 画像選択

- ユーザーは端末から画像を1枚選択できる。
- 選択UIはクリック操作とドラッグ＆ドロップ操作の両方を受け付ける。
- 選択した画像のプレビュー、ファイル名、ファイルサイズを表示する。
- 新しい画像を選択した場合は以前の選択を置き換える。

### FR-02 入力形式検証

- 許可するMIMEタイプは `image/jpeg`、`image/png`、`image/webp` とする。
- 最大ファイルサイズは10 MBとする。
- 拡張子だけでなく、MIMEタイプとファイルシグネチャを検証する。
- 無効な形式、空ファイル、サイズ超過をOpenAI APIへ送信する前に拒否する。
- 検証エラーは日本語でユーザーへ表示する。

### FR-03 説明レベル選択

ユーザーは次の3段階から説明の詳しさを選択できる。

| 値         | 表示名 | 期待する出力                                                         |
| ---------- | ------ | -------------------------------------------------------------------- |
| `brief`    | 短い   | 主要な被写体と状況を1～2文で説明                                    |
| `standard` | 標準   | 被写体、背景、構図、色、動作などを簡潔な段落で説明                   |
| `detailed` | 詳細   | 観察できる要素、位置関係、雰囲気、文字情報を複数段落または箇条書き化 |

初期値は `standard` とする。

### FR-04 画像説明生成

- フロントエンドは画像と説明レベルをFastAPIバックエンドへ送信する。
- バックエンドはPydanticAIを通じてOpenAI Responses API対応モデルを呼び出す。
- ローカル画像データはPydanticAIの `BinaryContent` としてモデルへ渡す。
- モデルの既定値は品質とコストのバランスを優先する `gpt-4.1-mini` とする。
- モデルIDは `OPENAI_MODEL` 環境変数で変更可能とする。
- 生成結果は日本語の画像説明テキストとして返す。
- 画像内の文字を確実に読み取れない場合は推測を事実として断定しない。
- 人物の身元、センシティブ属性、場所などを画像だけから根拠なく断定しない。

### FR-05 結果表示

- 処理中は進行状態を明示し、重複送信を防止する。
- 成功時は生成した説明を読みやすい領域へ表示する。
- 結果領域はスクリーンリーダーへ更新を通知する。
- ユーザーは生成結果をクリップボードへコピーできる。
- コピー成功・失敗の状態を画面上で通知する。

### FR-06 エラー処理

- 入力検証エラー、設定不足、OpenAI認証エラー、レート制限、タイムアウト、外部API障害を区別する。
- ユーザー向けエラーにはスタックトレース、内部パス、APIキー、プロバイダーの生レスポンスを含めない。
- 再試行可能なエラーでは、ユーザーに再試行可能であることを伝える。
- バックエンドは予期しない例外をグローバル例外ハンドラーで安全なレスポンスへ変換する。

### FR-07 ヘルスチェック

- バックエンドは `GET /health` を提供する。
- ヘルスチェックはOpenAI APIへの有料リクエストを行わない。
- 正常時はアプリケーション状態とバージョンを返す。

## 5. API要件

### POST `/api/v1/descriptions`

リクエストは `multipart/form-data` とする。

| パラメーター | 型       | 必須 | 制約                           |
| ------------ | -------- | ---- | ------------------------------ |
| `image`      | ファイル | はい | JPEG、PNG、WebP、最大10 MB     |
| `detail`     | 文字列   | はい | `brief`、`standard`、`detailed` |

成功レスポンスの概念スキーマ:

```json
{
  "description": "画像の説明",
  "detail": "standard",
  "model": "gpt-4.1-mini",
  "request_id": "uuid"
}
```

エラーレスポンスの概念スキーマ:

```json
{
  "error": {
    "code": "invalid_image_type",
    "message": "JPEG、PNG、WebP形式の画像を選択してください。",
    "request_id": "uuid"
  }
}
```

## 6. 非機能要件

### NFR-01 技術スタック

- Python 3.12.8を `.python-version` と `pyproject.toml` で固定する。
- Python依存関係はuvで管理し、`uv.lock` をコミット対象とする。
- バックエンドはFastAPI、Pydantic、Pydantic Settings、PydanticAIを使用する。
- フロントエンドはNext.js、React、TypeScriptを使用する。
- JavaScript依存関係はロックファイルで固定する。

### NFR-02 性能とタイムアウト

- 10 MBを超えるリクエストをOpenAI呼び出し前に拒否する。
- OpenAI呼び出しには構成可能なタイムアウトを設定する。
- 同一プロセスで無制限の同時外部API呼び出しを許可しない。
- ローカル利用を前提とし、通常の画像に対して結果待ち状態を明示する。

### NFR-03 セキュリティ

- `OPENAI_API_KEY` は環境変数からのみ読み込み、ソース管理へ含めない。
- `.env.example` にはダミー値または空値のみを記載する。
- CORSは既定で `http://localhost:3000` のみ許可する。
- API入力はPydanticモデルと明示的なファイル検証で検証する。
- APIにはローカル向けのレート制限を実装する。
- フロントエンドはCSP、HSTS、本番時の `nosniff`、フレーム拒否、Referrer Policyを設定する。
- ログに画像バイト列、生成説明全文、APIキー、Authorizationヘッダーを出力しない。
- 依存関係はロックし、脆弱性スキャン手順とSBOM生成手順を文書化する。

### NFR-04 プライバシー

- アプリケーション自身は画像または生成結果を永続保存しない。
- 画像はメモリ上で処理し、リクエスト終了後に参照を破棄する。
- ログへ画像内容や説明本文を記録しない。
- OpenAI APIへ送信されることを送信前に画面上で明示する。

### NFR-05 アクセシビリティ

- 全操作をキーボードで実行可能とする。
- フォーカス状態を明確に表示する。
- ファイル入力、詳細度、実行、コピー操作に適切なラベルを付ける。
- 状態とエラーは色だけに依存せず、テキストでも伝える。
- 十分な色コントラストとレスポンシブ表示を提供する。
- 画像プレビューの代替テキストは選択ファイルであることを説明する。

### NFR-06 可観測性

- バックエンドはJSON形式の構造化ログを出力する。
- ログ項目はタイムスタンプ、レベル、メッセージ、リクエストIDを含む。
- フロントエンドから受け取る値ではなく、サーバー側でリクエストIDを生成する。
- OpenAI呼び出しの成功、失敗、処理時間を機密情報なしで記録する。

### NFR-07 テスト

- バックエンドはpytestで単体テストとAPIテストを実装する。
- 外部OpenAI APIはテストでモックし、通常のテスト実行で課金を発生させない。
- フロントエンドはコンポーネントと主要ユーザーフローをテストする。
- E2E相当の統合確認ではバックエンドのOpenAIクライアントをスタブ化する。
- Property-Based Testingは部分適用とし、Hypothesisを使用する。
- PBT対象は入力正規化、列挙値、サイズ境界、APIスキーマの不変条件とする。
- Hypothesisの縮小機能を無効化せず、失敗時に再現情報を保持する。

### NFR-08 保守性

- API層、設定、画像検証、AIサービス、スキーマ、例外処理を分離する。
- フロントエンドはページ、アップロードUI、結果UI、APIクライアントを分離する。
- OpenAIモデルとタイムアウトは環境変数で変更可能とする。
- 公開関数と重要な設計判断へ簡潔なドキュメントを付ける。

## 7. ユーザーシナリオ

### 正常系

1. ユーザーが画像を選択する。
2. プレビューを確認する。
3. 説明レベルを選択する。
4. 画像がOpenAIへ送信されることを確認して生成を実行する。
5. 日本語説明を確認する。
6. 説明をクリップボードへコピーする。

### 入力エラー

1. ユーザーが未対応形式または10 MBを超えるファイルを選択する。
2. クライアント側で即座にエラーを表示する。
3. バックエンドも同じ制約を独立して検証し、不正入力を拒否する。

### 外部APIエラー

1. OpenAI APIがタイムアウト、認証失敗、レート制限、サービス障害を返す。
2. バックエンドは安全なアプリケーションエラーへ変換する。
3. フロントエンドは日本語のエラーと再試行可能性を表示する。

## 8. 対象外

- ユーザー登録、ログイン、権限管理
- データベース、サーバー側履歴、ブラウザ内履歴
- 複数画像の一括処理
- 画像URL入力
- 画像編集または画像生成
- OCR精度保証、人物識別、医療・法的判断
- クラウド配備用Infrastructure as Code
- 本番SLA、RTO、RPOの保証

## 9. 受け入れ基準

- JPEG、PNG、WebPの10 MB以下の画像を送信して日本語説明を取得できる。
- 短い、標準、詳細の選択によってプロンプトと期待する説明量が変わる。
- 未対応形式、空ファイル、10 MB超過をOpenAI呼び出し前に拒否できる。
- OpenAI APIキーが未設定の場合、安全で理解可能なエラーを表示できる。
- OpenAI APIをモックしたバックエンドテストが成功する。
- キーボードのみで画像選択、レベル選択、生成、コピーを操作できる。
- 画像、説明、APIキーがログや永続ストレージへ保存されない。
- Pythonはuvにより3.12.8で実行できる。
- バックエンドとフロントエンドのlint、型検査、テストが成功する。

## 10. 前提と決定

- 未記入だった質問には、明示されていた推奨選択肢を適用した。
- セキュリティベースラインを有効化する。
- レジリエンシーベースラインはローカル用途のため無効化する。
- Property-Based Testingは部分適用する。
- OpenAIモデルは `gpt-4.1-mini` を既定値とし、環境変数で変更可能にする。
